How scoreAesthetic() Works
A plain-language walkthrough of the function that scores every glaze palette in BklynClay, and a guide for adding new taste dimensions — like stripes vs. linear banding, gloss vs. matte, or crawling vs. smooth — without turning the system into a black box.
What scoreAesthetic Does

scoreAesthetic(glazes, weights) in scoring.js takes a palette (an array of 2+ glazes) and returns a single number from 0–100. That number drives sort order in Discover, the "Aesthetic score" badge on every card, and ranking inside the Calibrate comparison flow.

It is not a neural net and it has no training data baked in. It is seven simple, named measurements — each one a deliberate guess about what makes a palette feel intentional rather than random — combined with a weighted sum. Every number it produces can be traced back to a specific, readable rule.

Where to look: the whole implementation is ~50 lines in scoring.js — featureVector(glazes) computes the seven raw features, scoreAesthetic(glazes, weights) combines them.
The Seven Features

Each feature is normalized to a 0–1 range before weighting. Here's what each one is actually measuring and why it exists:

#FeatureWhat it measuresIntuition
f1Saturation spreadStandard deviation of saturation across the palette's glazesA palette that's all-vivid or all-muted feels flat; mixing intensities feels composed
f2Lightness rangeMax lum − min lum across the paletteToo tight a lightness range looks muddy on a pot; too wide can look disjointed. A sweet spot in the middle scores highest
f3Lightness balanceHow close the palette's average lightness is to a mid-tone (0.47)Palettes skewed all-dark or all-pale read as heavy or washed-out
f4Min pairwise distanceThe closest two glazes get to each other in hue/lum/sat spaceIf two glazes are nearly identical, the palette reads as a mistake, not a choice — this is the harshest penalty in the score
f5Hue harmonyBest fit against analogous, complementary, and triadic hue relationshipsBorrows directly from color theory — palettes that loosely match a known harmony pattern read as "designed"
f6Material varietyWhether finishes span more than one type (e.g. gloss + matte, not gloss + gloss)Visual texture variety reads as more considered than uniform finish
f7Achromatic penaltyPenalizes having more than one near-gray (low saturation) glazeOne neutral anchors a palette; two or more grays reads as indecisive rather than intentional
How Weights Combine

The default weights (used until a user calibrates their own) are fixed constants chosen by hand, not learned:

f4 — min pairwise distance
25%
f1 — saturation spread
18%
f3 — lightness balance
17%
f2 — lightness range
15%
f7 — achromatic penalty
13%
f5 — hue harmony
7%
f6 — material variety
5%
score = round(100 * (f1·w1 + f2·w2 + f3·w3 + f4·w4 + f5·w5 + f6·w6 + f7·w7))

Because it's a weighted sum of named, bounded features — not a model with millions of opaque parameters — every score is explainable: you can always say "this palette scored low because two glazes were nearly identical (f4) and it had three different grays (f7)."

Why Not Just Use AI?

A neural net or LLM judge could absolutely produce a "does this look good" score. The reason BklynClay doesn't work that way:

  • Traceability. Every score decomposes into seven readable numbers. If a palette scores 40, you can see exactly which feature dragged it down — instead of "the model said so."
  • No training data needed. The defaults work on day one with zero user history, because they encode established design rules (color harmony, contrast, balance) rather than learned correlations.
  • Personalizable without retraining a model. Phase 2 (below) tunes the same seven weights to an individual's taste using ~25 comparisons and a tiny logistic regression — fast, local, and auditable.
Phase 2: Learned Weights

The Calibrate flow shows pairs of saved palettes and asks "which do you prefer?" Each answer is stored as a comparison. After enough pairs, fitWeights() runs a small logistic regression (Bradley-Terry style) over the feature differences between the two palettes in each pair, and produces a personalized 7-weight vector saved to learnedWeights in localStorage.

Crucially, this still produces exactly the same kind of weighted sum over the same seven named features — calibration changes how much each feature matters to you, not what is being measured. The "Personalized score" badge always traces back to the same seven readable numbers.

Adding New Preferences (Stripes vs. Linear, Gloss vs. Matte, Crawling vs. Not)

All three of these are the same kind of change: a new feature dimension that the current seven don't capture, because they all describe color relationships, not finish/pattern relationships. The path to adding any of them is the same three steps:

1. Define what's measurable about it

Each glaze in glazes-data.js already carries a fin (finish) field — values like gloss, matte, transparent, crawl-*. That's enough raw material for several new features without touching the data model:

PreferenceNew feature, f8...What it would measure
Gloss vs. mattef8: finishContrastFraction of the palette that is gloss vs. matte — e.g. score high for a palette that's deliberately all-one or a clean 50/50 split, score low for an unintentional 2-gloss-1-matte mix
Crawling vs. smoothf9: crawlRatioSame shape as f7's achromatic penalty: does the palette use 0, 1, or 2+ crawl-texture glazes, and does that match the user's preference for "one textural accent" vs. "all-crawl" vs. "no texture"
Stripe vs. linear bandingf10: patternIntentThis one isn't a glaze property — it's how the glazes are applied. Needs a new field on the project/palette object (e.g. applicationPattern: 'stripe'|'linear'|'dip'|'random') set when a user builds a layout, not derived from GLAZES data
Key distinction: gloss/matte and crawling are glaze properties — already in GLAZES, so they slot into featureVector() immediately. Stripe vs. linear is an application property — it describes how glazes are arranged on a piece, which doesn't exist as data yet. That one needs a small data model addition before it can be scored at all.

2. Add it to featureVector()

Each new feature is just another 0–1 number appended to the array featureVector() returns. For example, gloss/matte balance:

const finishCounts = { gloss: 0, matte: 0 };
glazes.forEach(g => {
  if (g.fin === 'gloss' || g.fin === 'transparent') finishCounts.gloss++;
  if (g.fin === 'matte') finishCounts.matte++;
});
// 1.0 = clean split or clean uniform, lower = accidental in-between mix
const f8 = Math.abs(finishCounts.gloss - finishCounts.matte) / glazes.length;

Then add a default weight (e.g. start small, like 0.05, and rebalance the others so they still sum to 1) and update the array destructure everywhere featureVector() is consumed.

3. Let calibration tune it per-user

Because fitWeights() already runs a generic gradient descent over however many features featureVector() returns, a new feature is picked up automatically once it's added — no changes needed to the learning code itself. If someone consistently prefers matte-only palettes over mixed-finish ones during Calibrate, the regression will push that feature's learned weight toward what they showed, not what was hardcoded.

This is the point of the architecture: taste dimensions are added as named, inspectable features, and personalization happens by re-weighting those names — never by replacing the explainable model with an opaque one.
Worked Example: Stripe vs. Linear Banding

This is the most involved of the three because, as noted above, "stripe vs. linear" isn't a fact about a glaze — it's a fact about how a project's layers are arranged. Concretely:

  1. Add the field. When a project is saved (see persistence.js), record how the user applied the palette: { pattern: 'stripe' | 'linear-band' | 'dip' | 'random' }. Default to 'random' for existing saved projects so nothing breaks retroactively.
  2. Score the match, not the pattern itself. Patterns aren't inherently better or worse — what should be scored is whether the palette's structure fits the chosen pattern. E.g. for stripe, high-contrast lightness steps between adjacent colors read well; for linear-band, smooth lightness progression (sorted lums with small deltas) reads well. So f10 would be computed differently depending on the project's stored pattern.
  3. Surface it in Calibrate. Add stripe vs. linear example pairs (same palette, two different layouts) to the comparison pool so the regression has signal to learn from specifically on this axis, not just inherited from color-only pairs.

The output is still one number, 0–100, still fully attributable to seven-to-ten named features — just with one more axis of taste the user can teach the app about them.
